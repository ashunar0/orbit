## 1. 思想の土台: 人間の最終責任

現在の philosophy.md は「AI が書いて、人間が読む」を出発点にしている。
しかし議論の結果、これは **時代のスナップショット** であって、思想の核ではないと整理された。

### 核心

> コードの最終責任は人間にある。AIがどれだけ進化しても、これは変わらない。

- AI がレビューもする時代が来ても、プロダクトの責任を負うのは人間
- だから人間は「必要なときに読んで、正しさを判断できる」状態を維持しなければならない
- 法律のように、普段は読まなくても、読めば理解できる状態を保つ

### 「AI が書いて、人間が読む」の位置づけ

- 今この思想が **必要とされる理由の説明** であって、思想そのものではない
- 思想の核は **「読む側への最適化」**
- 読み手が人間でも AI でも、WHAT が明確で構造が予測可能なコードが最も価値がある

### 読む頻度が下がるほど規約の価値は上がる

```
毎日読む人:   文脈が頭にある → 多少バラついても追える
たまに読む人: 文脈がない → 構造が揃っていないと追えない
```

- AI に任せる範囲が広がれば、人間がコードを読む頻度は下がる
- 理解負債は「コードを読まない期間」に比例して溜まる
- 規約で構造が揃っていれば、間が空いても「前と同じパターンだ」と即座に復帰できる
- **規約は理解負債の利子をゼロにする仕組み**

## 2.「隠すな」の再定義 → 「隠すな、階層化しろ」

### 問題

philosophy.md は「隠さない」と書いているが、orbit-rpc は server.ts の関数を自動で Hono RPC に変換しており、**HOW は明らかに隠れている**。
「隠さない」の境界が言語化されていなかった。

### WHAT と HOW の区別

| 層 | 問い | 例 |
|---|---|---|
| **WHAT** | このコードは何をしているか？ | 「ユーザーを取得する」 |
| **HOW** | それはどう実現されているか？ | 「HTTP POST で /rpc/users/getUser に飛ぶ」 |

Orbit の server.ts は **WHAT は見える。HOW は隠れている**。

比較:

- **Rails**: `resources :users` → WHAT が隠れている（何が生えるか訳せない）→ NG
- **orbit-rpc**: `export async function getUser(id)` → WHAT は見える → OK
- **React Hook Form**: `register("name")` → WHAT が隠れている（何がバインドされるか訳せない）→ NG

### 新しい定義

> **「何をしているか（WHAT）」はコードに書いてあること。**
> **「どうやっているか（HOW）」は抽象化してよい。ただし境界を明示すること。**

- 「境界の明示」= `server.ts` というファイル名。「ここに書いた関数はサーバーで動く」ことをファイル名が示す
- HOW（HTTP 通信、Hono、RPC 変換）は知らなくていい

### HOW が隠れていても問題ない理由 — AI時代の分業

Orbit は「AI に書かせて、人間が読む」ことを前提としている。

- **人間の仕事**: コードを読んで、WHAT の正しさを判断する
- **AI の仕事**: コードを書く。壊れたら HOW のレベルで調査・修正する

HOW が壊れたときの対応は AI に任せられる。だから人間に見えるべきは WHAT だけでいい。

### 「隠さない」ではなく「階層化する」

実態に即した表現:

> **隠すな、階層化しろ**

コードの表面には WHAT だけを出す。HOW は規約の境界の裏側に置く。
読む人の理解度に応じて、必要な深さまで掘れるようにする。

```
Level 1: コードを読んで「何をしているか」がわかる → 誰でも
Level 2: 「なぜそう書くのか」がわかる → 規約を学べば
Level 3: 「裏で何が起きているか」がわかる → 掘れば
```

「全部見せる」でも「全部隠す」でもない。情報を階層化する。

```
全部隠す                    階層化                    全部見せる
Rails                      Orbit                   TanStack
resources :users     export function getUser()    useQuery({ queryKey,
                                                    queryFn, staleTime,
                                                    gcTime, retry, ... })
```

- **Rails 側**: 知ってる人は速い。知らない人は何が起きてるかわからない
- **TanStack 側**: 全部見える。でも毎回全部読む必要があり、本質が埋もれる
- **Orbit**: WHAT が見える。HOW は規約の裏側。知りたければ掘れる

## 3. Orbit の提供価値の構造

### 「Next.js でいいじゃん」問題

ライブラリとしての orbit-router / query / form / rpc は、機能的には既存の組み合わせで代替できる。
個々のパッケージに「これは他にない」という機能はない。

**ライブラリ単体で見たら、「Next.js でいいじゃん」は正しい。**

### 新規性は思想にある

Orbit が実際に提供しているのは、ライブラリ群ではなく **統合したときに生まれる一貫性**。
そして、その一貫性を支える **思想** に新規性がある。

### 3層の価値構造

```
思想（新規性がある。ライブラリ非依存）:
  │  規約駆動開発 (Convention-Driven Development)
  │  理解負債ゼロ
  │  WHAT/HOW の階層化
  │  Progressive Decomposition
  │
  │  → ブログ、ドキュメント、CLAUDE.md テンプレートとして
  │    Orbit を使わなくても適用できる
  │
統合の価値（Orbit だからこそ提供できる）:
  │  型一貫性: schema.ts → rpc → query → form が1つの型で繋がる
  │  規約一貫性: 全パッケージが同じ設計思想に従う
  │  フロー一貫性: State → Fetch → Transform → Mutate → Render
  │  React Compiler 互換: 思想から導かれる技術的帰結
  │
  │  → 「Next.js + バラバラのライブラリ」では
  │    型の再定義・規約のバラつきが避けられない
  │
ライブラリ群（個々は代替可能。統合に意味がある）:
      orbit-router / query / form / rpc
```

### 型一貫性 — 統合の具体的な価値

バラバラのスタック（TanStack Query + RHF + tRPC 等）では、型の再定義・変換がライブラリの接合点ごとに発生する。
変更があったら全部手動で修正が必要。これは **型レベルの理解負債**。

Orbit は統合されているため、schema.ts の Zod 定義が唯一の源泉として全パッケージに流れる:

```
schema.ts（唯一の源泉）
  → orbit-rpc: schema.ts の型がそのまま使われる
  → orbit-query: rpc の返り値型がそのまま流れる
  → orbit-form: schema.ts の型がそのまま流れる
```

schema.ts を変えれば全部追従する。**この型一貫性は統合されたエコシステムでしか実現できない。**

### 「Next.js でいいじゃん」への回答

> 思想は Next.js + 何でも に適用できる。CLAUDE.md に規約を書くだけでも効果がある。
>
> でも schema.ts から rpc → query → form まで型が一本で繋がるのは、
> 統合されたエコシステムでしかできない。
> バラバラのライブラリを組み合わせると、型の変換・再定義が至るところに挟まる。
> その1つ1つが理解負債になる。
>
> Orbit は思想と型一貫性を、ひとつのツールキットで提供する。

### 成功プロジェクトとの共通パターン

「思想 + ツール」は成功プロジェクトに共通する形態:

```
Rails    = 「Convention over Configuration」 + フルスタック FW
React    = 「UI はコンポーネントツリー」 + ライブラリ
Tailwind = 「Utility-First CSS」 + CSS フレームワーク
htmx     = 「Hypermedia as the engine」 + 軽量ライブラリ
Orbit    = 「Convention-Driven Development」 + 統合ツールキット
```

どれも思想が先にあって、ツールはその具現化。

## 4. 未解決の課題

### Progressive Decomposition の「分解後」

Progressive Decomposition は「分解していい」と言っているが、**分解のパターン** が未定義。

- hooks.ts が大きくなったら → hooks/ ディレクトリに分割？
- page.tsx が大きくなったら → _components/ に切り出し？
- 全体が大きくなったら → ルート自体を分割？

出発点は1つに収束するが、分解後にバラつくリスクがある。
実証実験の規模が小さいため、まだ検証できていない。

分解パターンも規約化するか、自由に任せるか — 判断が必要。

  → 「思想だけでも使えるが、統合するとこうなる」

   - WHAT/HOW の階層化

3. **統合の価値**: 「schema.ts を変えたら全部変わる — 型一貫性の話」
